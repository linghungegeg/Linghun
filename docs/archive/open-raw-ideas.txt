项目全链路规划手册：融合 CCB、OpenCode 与 Hermes 的下一代 AI 编程助手
一、项目定位与愿景
一句话定位：一款 低成本、可进化、开放架构 的 AI 编程助手，融合了 CCB（Claude Code Best）的极致缓存能力、OpenCode 的模型无关性与模块化架构、Hermes Agent 的跨会话记忆与技能进化机制。

核心价值：

越用越省钱：通过极致的 Prompt Cache（≥96% 命中率）将 API 成本降至市场最低。

越用越聪明：通过持久记忆与技能固化，AI 能记住项目偏好、自动沉淀可复用技能。

完全开放：支持任意模型（DeepSeek/Claude/GPT/本地模型），插件化架构，可深度集成 IDE。

4.2 核心模块设计
🧠 记忆与进化系统（借鉴 Hermes）
跨会话记忆：自动维护 MEMORY.md（项目知识）、USER.md（个人偏好），每次对话加载。

技能固化：成功解决问题的流程自动生成可复用的 Skill 文件，存储在 .skills/ 目录。

学习闭环：定期后台演练，整合新旧知识，优化技能库。

⚡ 执行与工具系统（借鉴 CCB）
原子化工具（参考 CCB 的 40+ 内置工具）：

Read（分页+行号）、Write、Edit（唯一性约束）

Glob、Grep（高性能搜索）

Bash（安全沙箱执行）

AgentTool（子代理派发）

权限分级：read / write / exec 三级控制，敏感操作需用户确认。

🛡️ 极致成本控制（CCB-Dev-Boost 核心）
三层 Prompt 结构：

静态层：系统指令 + 工具定义 → 永久缓存

半稳定层：CLAUDE.md / 项目规则 → 会话级缓存

动态层：用户查询 + 最近文件内容 → 每次拼接

智能缓存标记：cache_control 仅放在静态/半稳定层末尾，避免碎片化。

冷启动强制预热：命中率低于 80% 时报错，引导用户先建立索引。最好是每次提交完就要询问用户是否要更新索引

🔌 开放集成能力（借鉴 OpenCode）
模型无关：通过 Vercel AI SDK 或自研 Gateway 接入 DeepSeek、Claude、GPT、Ollama 等。

LSP 支持：启动时自动为各语言（Go, TS, Python…）拉起 LSP 客户端，提供语义补全、定义跳转。

8. AI 说“不”（能力边界诚实）
当前AI编程助手最常见的坑就是“什么都说能做”，导致新手花大量时间尝试一个根本不可能实现的功能。

明确拒绝机制：当用户提出的需求超出当前能力范围（如“帮我画一张图”、“分析这个图片里的代码”、“联网搜索最新文档”），AI必须明确回复“我无法做到这一点，原因是：xxx”，而不是编造或瞎猜，拒绝虚构无法完整的开发任务

提供替代方案：拒绝的同时，主动建议可行的替代路径。例如：

“我无法识别图片中的代码，但你可以将代码以文本形式粘贴给我，或者使用 Claude 3.5 Sonnet 模型（支持视觉）来尝试。”

能力清单自检：AI内置一份自己能力的精确清单，每次回答前自动核对。用户也可以通过 /capabilities 命令查看当前模型支持的所有能力（文本生成、文件读写、命令执行、图像理解……）。

模型切换建议：当用户需求需要特定能力（如识图）而当前模型不支持时，AI主动提示切换模型的命令：

“当前 DeepSeek V4 Pro 不支持图像识别。你可以运行 xxx model set claude-sonnet-4 切换到 Claude，然后重试。”

9. 多AI协作指挥（委派与融合）
单个模型总有短板（DeepSeek 便宜但不会识图，Claude 贵但代码强，GPT 擅长设计）。让多个AI各司其职，协同工作。

委派模式（Orchestrator）：主AI负责拆解任务，将子任务派发给最适合的模型执行，然后汇总结果。

示例工作流：

用户：“识别这张设计图并生成HTML代码。”
主AI（成本最低的模型）判断：识图 → 派发给 Claude 或 GPT-4V；代码生成 → 派发给 DeepSeek V4 Pro 或 Claude。
最终将两张结果合并输出。

模型路由器：配置文件定义每个任务的“最佳模型”：

yaml
routers:
  - capability: image_understanding
    model: claude-sonnet-4
  - capability: code_generation
    model: deepseek-v4-pro
  - capability: planning
    model: gpt-4o
并行调用：当一个任务不依赖顺序时（例如同时搜索三个文件、分析两个独立函数），可并行调用多个模型实例，最后合并结果。

成本透明：每次委派后显示各子任务的模型、Token 消耗、费用，让用户清楚成本分布。

10. 多工作流覆盖（能力互补，成本最优化）
利用不同模型的能力互补，实现“一个任务，多种工具配合”，而不是单模型死磕。

视觉+代码工作流（你提到的DeepSeek不能识图）：

当用户提供图片时，自动调用支持视觉的模型（Claude/GPT-4V）提取文字/布局信息。

将提取的结构化信息传给 DeepSeek 生成代码或执行后续任务。

综合成本远低于全程使用高价视觉模型。

搜索+生成工作流：

当用户问“最新的React 19有哪些特性”时，AI自动判断需要联网搜索。

调用内置的 web_search 工具（可接入免费或付费搜索API）获取最新资料。

将搜索结果作为上下文，再让模型生成最终答案。

多模态输入处理：用户可以通过 /image 命令上传图片，AI自动识别并提示切换模型；或通过 /url 读取网页内容。

工作流模板：预置常见多模型协作场景，用户可直接调用：

/workflow design-to-code → 图片 → 代码

/workflow doc-to-code → 文档 → 实现

/workflow bug-fix → 错误日志 → 修复代码

补充第11条：信息面板与交付顺序
11. 信息面板（Dashboard）显示日用关键信息
在终端界面（TUI）中，始终保留一个固定区域（如底部状态栏或右侧边栏），实时显示用户最关心的数据，避免频繁输入命令查询。

实时命中率：当前会话的缓存命中率（如 命中率: 96%），用颜色区分（绿 >90%，黄 70-90%，红 <70%）。

Token 与费用：本次会话累计输入/输出 Token、预估费用（如 ¥0.187），以及本次调用相比无缓存的节省金额（省 ¥3.42）。

当前模型：正在使用的模型名称（如 DeepSeek-V4-Pro），支持一键切换快捷键。

工具调用状态：最近一次工具调用（如 Read file.ts）的状态（成功/失败/执行中）。

快捷指令提示：常用命令（/help, /stats, /model set, /learn）的快捷键提示。

低命中率告警：当命中率低于80%时，面板自动高亮警告并建议“执行 /warmup 预热”。

12. 交付顺序：先终端（TUI），后客户端（GUI）
优先交付终端版本，验证核心价值；待用户积累后再开发图形界面。

Phase 1 (MVP) - 终端版本：

基于 Ink (React for Terminal) 或 Blessed 构建交互式 TUI。

实现上述信息面板的所有功能。

支持 Windows Terminal、macOS Terminal、iTerm2 等主流终端。

Phase 2 (后续) - 客户端版本：

使用 Tauri 或 Electron 打包成跨平台桌面应用。

保留 TUI 的核心交互，增加图形化配置界面、历史会话管理。

提供系统托盘、全局快捷键等增强功能。
